--------------------------------------------------------------------------------
1. WHAT THIS IS
--------------------------------------------------------------------------------
An interactive, map-based web app showing how climate change is projected to
reshape England's surf breaks. Users click a break on a map of England and see a
profile panel with climate-risk indicators for that specific spot.

Audiences (from PRD): surfers, surf-related business owners, environmental
activists. Tone: non-alarmist, honest about uncertainty, does NOT predict wave
quality — only projected coastal/ocean conditions.

V1 SCOPE FOR THIS BUILD (important):
  * Sea level rise            -> IMPLEMENTED
  * Coastal erosion risk      -> IMPLEMENTED (incl. recession-zone map)
  * Sea SURFACE TEMPERATURE   -> DEFERRED at the user's explicit request.
                                 Not implemented. (PRD section 4.4 / NWSPPE.)
England only. 22 curated breaks.

--------------------------------------------------------------------------------
2. ARCHITECTURE (how it fits together)
--------------------------------------------------------------------------------
Pattern (from PRD): pre-process once -> static JSON -> dependency-free frontend.
No API calls are made at runtime by the website.

  surf_breaks.csv      Seed: 22 breaks. Columns: id,name,latitude,longitude,region
  config.py            Endpoints, chosen scenarios, HTTP/retry + tuning constants
  preprocess.py        Queries the two live APIs -> writes data/surf_breaks.json
  data/surf_breaks.json  Generated artefact the frontend loads (~320 KB)
  web/index.html       Map + profile-panel shell
  web/style.css        Ocean palette, mobile-first (desktop side drawer ~33vw;
                       mobile bottom sheet)
  web/app.js           Leaflet map, markers, panel, sea-level SVG chart, and the
                       erosion recession inset map
  README.md            Run instructions + limitations
  requirements.txt     Only `requests` (frontend has no build step)

RUN:
  pip install -r requirements.txt
  python preprocess.py            # all 22 breaks (SLOW ~15 min, see quirks)
  python preprocess.py --limit 3  # quick smoke test
  python -m http.server 8000      # then open http://localhost:8000/web/

Frontend dependencies are CDN Leaflet 1.9.4 + CARTO "light_all" basemap tiles.
No API key, no npm, no build.

3. DATA SOURCES (exact) AND SCENARIOS CHOSEN
--------------------------------------------------------------------------------
All sources are UK public open data under Open Government Licence v3.0. No secrets
or keys required.

(A) SEA LEVEL RISE — Met Office UKCP18, ArcGIS FeatureServer
    URL: https://services.arcgis.com/Lq3V5RFuTBC9I7kv/arcgis/rest/services/
         Sea_Level_2010_2100/FeatureServer/1
    Polygon layer; each polygon ~12 km of coastline.
    Fields: seaLevelAnom_2010 ... seaLevelAnom_2100 (decadal, cm, anomaly vs the
            1981-2000 baseline) + rcp_percentile (string).
    CHOSEN SCENARIO: RCP8.5, 50th percentile. Stored value is lowercase
            "rcp85_50" (NOT "RCP85_50th"). This was a PRD-decided default.
    Full distribution available if needed later: rcp{26,45,85}_{5,50,95}.

(B) COASTAL EROSION — NCERM 2024, Environment Agency, OGC API - Features
    BASE: https://environment.data.gov.uk/spatialdata/ncern-national-2024/
          ogc/features/v1
    NOTE THE SPELLING: the path is "ncern" (a typo baked into the official URL),
    NOT "ncerm". ncerm-* returns 404.
    Collections are named NCERM_<scenario>_<year>_<allowance>CC, e.g.:
        NCERM_SMP_2055_70CC, NCERM_SMP_2105_70CC,
        NCERM_NFI_2055_70CC, NCERM_NFI_2105_70CC
      - SMP = "with Shoreline Management Plan delivered" (the realistic/planned
              scenario; defences held where the plan says so)
      - NFI = "No Future Intervention" (counterfactual: defences not maintained)
      - 2055 = Medium Term; 2105 = Long Term
      - allowance CC: 0CC = present-day climate (2020),
                      70CC = Higher Central (UKCP18 RCP8.5 70th pct),
                      95CC = Upper End (RCP8.5 95th pct)
    CHOSEN ALLOWANCE: 70CC (Higher Central), for consistency with the RCP8.5
            sea-level default. (config.EROSION_CC = "70")

    Key feature properties (per "frontage" = a coastline unit):
        frontageid, smp_name (policy-unit name), smp_pu (policy-unit code),
        mt_smp / lt_smp        = SMP policy text, medium / long term
        mt_smp_int / lt_smp_int= interpretation text, medium / long term
        smp2055_70 / smp2105_70 / nfi2055_70 / nfi2105_70 = a numeric "band"
    Observed categorical values:
        policy:        "Hold The Line", "Managed Realignment",
                       "No Active Intervention", (blank)
        interpretation:"Erosion restricted", "Erosion unrestricted",
                       "Stop Maintaining", (blank)

    IMPORTANT — the numeric "band" field is UNDOCUMENTED. It equals "maxoverlap"
    in many frontages and ranges widely (0 to ~250). DO NOT present it as
    "metres of recession" — that would be false precision. We surface it only as
    a relative "band" indicator. The honest recession visual is the zone POLYGON
    footprint (see section 4D).

--------------------------------------------------------------------------------
4. CALCULATIONS / METHODS (the important bit)
--------------------------------------------------------------------------------
(A) SEA LEVEL per break  (preprocess.fetch_sea_level)
    - Breaks sit just INSHORE of the offshore sea-level polygons, so a plain
      point-in-polygon query returns NOTHING. We buffer the point outward via the
      ArcGIS `distance` parameter, trying radii [2000, 6000, 12000, 25000] m and
      stopping at the first that returns polygons (config.SEA_LEVEL_BUFFERS_M).
    - WHERE clause filters rcp_percentile='rcp85_50'.
    - If multiple polygons fall within the buffer, we AVERAGE each decadal field
      across them (they're near-identical neighbours).
    - Output: sea_level.anomaly_cm = {"2030":.., ... "2100":..} in cm,
      plus status, scenario, baseline, n_polygons, buffer_m.
    - Sanity: ~17 cm by 2030 rising to ~81 cm by 2100 for Cornwall — consistent
      with UKCP18 RCP8.5 literature.

(B) NEAREST-FRONTAGE MATCHING (erosion)  (preprocess._fetch_nearest_frontage)
    - OGC API needs an `Accept: application/json` header (the `f=json` query
      param returns HTML). Spatial filter is via `bbox` in CRS84 (lon/lat).
    - We query a bbox around the break, trying half-widths
      [0.02, 0.05, 0.1, 0.2] degrees (config.EROSION_BBOX_DELTAS), stopping at the
      first that returns features.
    - Among returned frontages we pick the NEAREST by minimum great-circle
      (haversine) distance from the break to any polygon vertex. (No shapely /
      pyproj — pure Python; ample at this scale.)
    - If nearest > EROSION_MAX_DISTANCE_M (2000 m) the break is reported as
      "no_nearby_frontage" rather than risk-free. This is the V1 answer to PRD
      open question #2 (not-at-risk vs data gap).

(C) FOUR SCENARIO/TERM COMBOS stored per break (preprocess.fetch_erosion):
        erosion.medium_term_with_smp        {policy, interpretation, band, zone}
        erosion.long_term_with_smp          {policy, interpretation, band, zone}
        erosion.medium_term_no_intervention {band, zone}
        erosion.long_term_no_intervention   {band, zone}
    Plus erosion.smp_name, smp_pu, climate_allowance, nearest_frontage_m, status,
    risk_level. (SMP collections carry the policy/interpretation text; NFI
    collections carry only the band.)

(D) RECESSION-ZONE POLYGONS (the inset map):
    - Each NCERM frontage is a genuine SOURCE UNIT of coastline (consistent
      cliff/defence characteristics). We pull that whole polygon as-is. We do NOT
      clip it. We currently store the SINGLE nearest frontage per scenario.
      (We considered merging neighbouring frontages within a radius for more
      coastal context but reverted it — the single source unit is faithful, and
      the inset's max-zoom cap handles the "too tight" feel instead.)
    - Geometry is SIMPLIFIED before storage: pure-Python Ramer-Douglas-Peucker
      at eps = 0.00003 deg (~3 m), coords rounded to 5 dp, z dropped, degenerate
      rings removed. This took the JSON from ~7 MB to ~320 KB with no visible
      change at the inset's scale. (preprocess._clean_geometry / _rdp.)
    - Zone dimensions are real (e.g. Cromer ~450x130 m, Fistral ~210x210 m), not
      slivers.

(E) MARKER / RISK COLOUR  (preprocess._risk_level):
    Derived ONLY from the LONG-TERM (to 2105) with-SMP policy/interpretation:
        high     <- interpretation "Erosion unrestricted"  OR policy "No Active Intervention"
        moderate <- interpretation "Stop Maintaining"      OR policy "Managed Realignment"
        low      <- interpretation "Erosion restricted"    OR policy "Hold The Line"
        unknown  <- otherwise
    CAVEAT 1: this measures MANAGEMENT (is the coast defended?), NOT erosion
              magnitude. "Hold The Line" reads low because it's defended, even if
              underlying erosion pressure is high.
    CAVEAT 2: the "high" branch is tested first, so when policy and interpretation
              disagree, the interpretation effectively wins. Real example: Fistral
              is policy "Managed Realignment" (would be moderate) but
              interpretation "Erosion unrestricted" -> classed HIGH. This
              precedence is currently implicit, not a deliberate decision — a
              candidate to revisit (see section 8).

5. FRONTEND BEHAVIOUR & GOTCHAS
--------------------------------------------------------------------------------
- Map: Leaflet, fits to all 22 markers. Markers are circleMarkers colour-coded by
  erosion risk_level (red=high, amber=moderate, teal=low, grey=unknown).
- Click a marker -> profile panel. Desktop = right side drawer (width 33vw,
  min 380 / max 620 px). Mobile = bottom sheet.
- Panel contents: identity, Coastal Erosion Risk (pill + policy table + recession
  inset map), Sea Level Rise (plain-English + inline SVG time-series chart + table),
  and a fixed epistemic-honesty disclaimer.
- Recession inset map: with-SMP zones filled by default (amber = to 2055, red =
  to 2105). A "show 'no defences' outline" checkbox overlays the No-Intervention
  zones as dashed outlines. Scale bar included. fitBounds uses maxZoom:15 + pad
  0.35 so short frontages keep coastal context.
- DEEP LINKS (also handy for testing/screenshots):
      ?break=<id>        opens that break's panel on load
      ?break=<id>&nfi=1  also switches on the "no defences" outline

  LEAFLET GOTCHA (cost us real time — do not regress):
  A Leaflet map MUST have a view set (setView/fitBounds) BEFORE you add vector
  layers, or the SVG renderer never initialises and the polygons silently never
  paint (overlay pane stays empty; getBounds still works, so it's confusing).
  buildErosionInset() therefore calls insetMap.setView(...) immediately after
  L.map(), before adding tiles/zones; fit() refines later. Also: invalidateSize()
  must run before fitBounds when the panel is mid-transition, and destroyInset()
  is wrapped in try/catch so a bad inset can never block break-switching.

--------------------------------------------------------------------------------
6. DATA-SOURCE QUIRKS (save yourself the rediscovery)
--------------------------------------------------------------------------------
- NCERM endpoint is misspelled "ncern-national-2024" (official).
- NCERM/Environment-Agency server is SLOW and FLAKY (intermittent HTTP 504).
  Every request retries with backoff (config.HTTP_RETRIES / HTTP_BACKOFF_S).
  A full 22-break run takes ~15 minutes. The pipeline NEVER fabricates data: if
  an indicator can't be fetched after retries it's recorded with a non-"ok"
  status and null value, and the run continues. Last full run: 22/22 ok on both
  indicators (no fallbacks were needed).
- OGC API: use header Accept: application/json (f=json returns HTML). bbox in
  lon,lat (CRS84). storageCrs is EPSG:4326. Coordinates come as [lon,lat,z]; we
  drop z.
- ArcGIS: rcp_percentile values are lowercase ("rcp85_50"). Point intersect needs
  a distance buffer (breaks are inshore of the polygons).
- The numeric erosion band == maxoverlap in places; undocumented; not metres.
